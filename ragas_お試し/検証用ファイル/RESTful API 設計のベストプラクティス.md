# RESTful API 設計のベストプラクティス
このドキュメントは、Webサービスの開発において統一性と保守性を高めるためのRESTful API設計の原則とベストプラクティスを定義します。

---
## 1. URI (リソース) 設計
- **原則**: URIは**リソース（資源）**を表現し、**名詞**を使う。動詞（例: `getProducts`）は使用しない。
- **良い例**: `/api/v1/products`, `/api/v1/users/123`
- **悪い例**: `/api/v1/getProductList`, `/api/v1/userProfile`

### ✅ 階層構造
- リソース間の関係はURIで階層的に表現します。
- 例: あるユーザーの注文リストを取得する場合: `/users/{userId}/orders`

---
## 2. HTTPメソッドの活用
- **原則**: リソースに対する操作は、HTTPメソッド（動詞）にマッピングする。

| HTTPメソッド | 目的 | 安全性/冪等性 |
|---|---|---|
| **GET** | リソースの**取得** | 安全（変更なし）かつ冪等（何度実行しても同じ結果） |
| **POST** | 新しいリソースの**作成** | 安全でも冪等でもない |
| **PUT** | リソースの**完全な更新**または作成 | 冪等（何度実行しても同じ結果）だが安全ではない |
| **DELETE** | リソースの**削除** | 冪等（何度実行しても同じ結果）だが安全ではない |

---
## 3. ステータスコードの利用
- HTTPステータスコードを使用して、APIリクエストの結果を正確に伝えます。

| ステータスコード | 分類 | 用途 |
|---|---|---|
| **200 OK** | 成功 | GET, PUT, DELETEなど成功時の一般的な応答 |
| **201 Created** | 成功 | **POST**リクエストで新しいリソースが正常に作成されたとき |
| **400 Bad Request** | クライアントエラー | リクエストボディ/パラメータの形式が不正な場合 |
| **401 Unauthorized** | クライアントエラー | 認証情報（トークンなど）が不足または無効な場合 |
| **404 Not Found** | クライアントエラー | リソースが見つからない場合 |
